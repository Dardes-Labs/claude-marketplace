# TaDa — companion plugin

Companion Claude Code plugin for **TaDa**, a per-dose medication-adherence system. The core is the TaDa **iOS app**; this plugin is the Mac surface. It reads the per-dose `state.json` the app syncs through iCloud Drive, surfaces today's **missed** and **still-to-take** doses right in the chat at session start, and lets you mark a dose **Taken** or **Skipped** from the Mac.

Built because ADHD + "just remember to take it" don't mix.

The plugin is **config-less and app-dependent**, and **never touches HealthKit** — `state.json` is its only surface, and the iOS app is the sole writer to Apple Health. Without the app there's nothing to read, so the plugin stays quiet. Design source of truth: the PRD in [`Dardes-Labs/tada-ios`](https://github.com/Dardes-Labs/tada-ios) → `.claude/prds/tada-medication-sync.md`.

## Install

```
/plugin marketplace add Dardes-Labs/claude-marketplace
/plugin install tada-medication-reminder@dardes
```

You also need the **TaDa iOS app**, signed into the same iCloud account — it's what produces `state.json`.

## How it works

```
SessionStart hook
   └── python3 scripts/tada.py session-context
         reads  iCloud~es~dard~tada/Documents/state.json  (the app's iCloud container)
         └── prints one JSON object Claude Code reads on exit 0:
               • systemMessage     → today's missed + still-to-take doses, shown to you in chat
               • additionalContext → the same state to Claude, so it can log a dose on request
             (no data, or file date ≠ today → silent; never nags from a stale day)

Log from the Mac
   /tada take "Adderall"   ·   /tada skip "Multivitamin"   ·   /tada undo "Adderall"
   └── writes status + logged_at + source:"plugin" back to state.json (atomic)
         └── the iOS app mirrors the dose into Apple Health (HKMedicationDoseEvent)
```

At session start the hook prints a short summary — **missed** doses (overdue and unlogged) and the ones **still to take** later today — which Claude Code shows you in the chat through `systemMessage`. When every scheduled dose for today is already logged, it says so; when there's no data for today (iCloud not synced, or the file isn't today's) it stays silent. It reads one small file synchronously, so the session isn't delayed — and a mid-session `/compact` doesn't re-show what you already saw when the session opened. No macOS system notification is involved.

## Logging from the Mac

Use the `/tada` slash command:

| Command | Does |
| --- | --- |
| `/tada` or `/tada status` | List today's doses and status (✓ taken · ✗ skipped · ○ pending). |
| `/tada take <med>` | Mark the **earliest pending** dose Taken. |
| `/tada skip <med>` | Mark the earliest pending dose Skipped. |
| `/tada undo <med>` | Reset that med's most recently logged dose to pending. |

Add `--at HH:MM` to target a specific dose of a multi-dose medication (e.g. `/tada take Adderall --at 20:00`), or `--at prn` to log an as-needed dose. A medication can be named by its display name or its `id`. The same engine backs the hook and the command — `scripts/tada.py` (stdlib Python 3, no dependencies).

Marking writes `source: "plugin"`; the iOS app picks that up and mirrors it into Apple Health. The plugin itself never writes HealthKit. If today's `state.json` hasn't been published yet (its `date` isn't today), the plugin refuses to log against the stale file and tells you to log on the phone.

## The sync contract

`state.json` lives in the iOS app's own iCloud container at `~/Library/Mobile Documents/iCloud~es~dard~tada/Documents/state.json`, read **and** written by both surfaces. Each dose is keyed `(medication id, date, scheduled_at)`; `status` ∈ `taken` / `skipped` / `pending`; conflicts resolve by **last-write-wins on `logged_at`**; `source` (`ios-app` / `widget` / `plugin` / `apple-health`) tells the app what to mirror and prevents loops. The file describes a **single local day** and is regenerated by the app at rollover — a file whose `date` isn't today is treated as not-yet-published. Full versioned schema (`schema_version: "1.0"`): FR-5/FR-6 of the PRD.

## Why the Mac plugin never touches HealthKit

By design the **iOS app is the sole HealthKit writer**; the Mac plugin works only through `state.json`. That isn't a stopgap — HealthKit on macOS is doubly gated (verified 2026-04-30):

1. **Capability gating.** HealthKit isn't in [Apple's macOS supported-capabilities list](https://developer.apple.com/help/account/reference/supported-capabilities-macos/) — for a macOS App ID the standard capability checkbox is disabled. The only path is an Apple-reviewed **Capability Request**, granted per-app and not guaranteed.
2. **Signing / AMFI.** `com.apple.developer.healthkit` is a *restricted* entitlement; macOS `amfid` SIGKILLs ad-hoc-signed binaries that carry it — so it needs a fully provisioned, signed app, not a hook-invoked CLI.

The medication types do exist in the macOS 26 SDK (`HKMedicationDoseEvent`, `HKMedicationConcept`, `HKUserAnnotatedMedication`) — the framework is there, only the entitlement is gated. On iOS both entitlements are self-service, which is why the app owns the HealthKit side.

## Environment variables

| Var | Default | Purpose |
| --- | --- | --- |
| `MEDICATION_ICLOUD_DIR` | `~/Library/Mobile Documents/iCloud~es~dard~tada/Documents` | Folder holding `state.json` (the app's iCloud container) |
